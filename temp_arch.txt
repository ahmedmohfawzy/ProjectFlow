تقرير المراجعة المعمارية
‏ProjectFlow™ — Solution Architect Review
إعداد: Ahmed M. Fawzy  |  التاريخ: 16 أبريل 2026  |  الإصدار: 2.0

هذا التقرير يقدم مراجعة فنية شاملة لتطبيق ProjectFlow من منظور Solution Architect و Senior Software Engineer، ويُحدد المشكلات القائمة، الفجوات المعمارية، و الخطوات المطلوبة لتحويل التطبيق من نموذج أولي (Prototype) غني بالخصائص إلى منتج مؤسسي متكامل مع Microsoft Planner و Dynamics 365 Project Accounting، يستخدمه جميع موظفي الشركة.

١. الملخص التنفيذي (Executive Summary)
التطبيق في وضعه الحالي يمثّل نموذجاً أولياً قوياً وغنياً بالخصائص — يحتوي على أكثر من 17,400 سطر من JavaScript موزّعة على 24 module، يغطي Gantt و CPM و EVM و Dashboard و Reports (PDF/Excel) و Resource Management و Network Diagram و Kanban و Multi-Project Portfolio و PWA. غير أن هذه الخصائص بُنيت بسرعة كبيرة أدت إلى تراكم دَين تقني (Technical Debt) واضح:
ملف app.js وحده يتجاوز 5,767 سطر، و reports.js يتجاوز 2,285 سطر — Monolith غير قابل للاختبار أو الصيانة على المدى الطويل.
تم رصد أكثر من 90 مشكلة أمنية/خوارزمية/أداء في المراجعات السابقة، وبعضها حرج (XSS، Directory Traversal، Critical Path Bugs، TCPI Division by Zero).
التكامل مع Microsoft Planner متقدم بنسبة ~70% (قراءة وكتابة مهام)، لكن لا يدعم Buckets creation و Attachments و Checklists push.
التكامل مع D365 Project Accounting متقدم بنسبة ~40% (قراءة غالباً)، ويغيب عنه Timesheets, Expenses, Invoicing, WBS push-back, Project-level updates.
البنية التحتية الحالية (Browser + IndexedDB + Node/SQLite المحلي) لا تصلح لنشر مؤسسي لجميع الموظفين — لا يوجد Multi-tenant، لا Central Auth، لا Real-time Collaboration، لا Audit Trail مركزي.
التوصية الرئيسية: إعادة تشكيل المشروع على شكل منتج مؤسسي (Enterprise SaaS) يعمل داخل Microsoft 365 — كتطبيق Teams App مع Azure Backend — ومتبقٍ على واجهة المستخدم الحالية كـ Web UI أساسية. يركز العمل في المرحلة الحالية على تكامل ثنائي الاتجاه (Two-way Sync) مع MS Planner و D365 Project Accounting، مقسَّماً على 12 سبرنت أسبوعي.

٢. المعمارية الحالية (Current Architecture)
٢.١ نظرة عامة على الطبقات
التطبيق مبني بمعمارية ثلاثية الطبقات (Client-heavy) مع احتياطات متعددة للتخزين:
‏Presentation: HTML/CSS + HTML5 Canvas (Gantt, Network, EVM, Dashboard). ملف واحد index.html يحتوي جميع الـ views.
‏Application: 24 JS module تعمل جميعها كـ IIFE على window.PF (Global). لا يوجد Build system و لا Module bundler.
‏Data: ثلاثية التخزين: localStorage → IndexedDB (Dexie) → SQLite عبر Express/Node.js (server.js).
‏Integrations: ms-graph.js (MSAL + Planner) + d365.js (Dataverse + F&O OData) — تعمل مباشرة من المتصفح.
‏Offline: Service Worker (sw.js) + Web App Manifest (PWA).
‏Optional wrappers: Tauri config (Desktop) + Plugins system.
٢.٢ مخطط التدفق (Data Flow)
‏ UI (app.js)  ⇅  EventBus / StateManager               ⇅ Storage Adapter  →  [ IndexedDB (Dexie) | localStorage | SQLite (server.js) ]               ⇅ MSAL Auth  →  [ Microsoft Graph (/planner/*) | D365 OData (/data/Projects) ] 
• المتصفح يتولى بنفسه مصادقة MSAL، و استدعاء Graph/D365 APIs مباشرة دون backend proxy. هذا يُعد Anti-pattern لأي نشر مؤسسي لأن: (1) Tokens تعيش في المتصفح وتفتح سطح هجوم أكبر. (2) لا يوجد مكان مركزي لتطبيق سياسات (Rate limiting, Throttling, Auditing). (3) لا يمكن تطبيق Conditional Access أو IP restrictions على الـ backend.
٢.٣ نقاط القوة (Strengths)
تغطية وظيفية واسعة تنافس Microsoft Project في قائمة الخصائص الأساسية.
استخدام MSAL مع PKCE بدل Client Secrets — قرار سليم لتطبيق Single-Page App.
‏EventBus + StateManager خطوة جيدة نحو فصل المسؤوليات.
‏Canvas rendering يتيح أداءً جيداً بدون Framework ثقيل.
‏PWA مع Service Worker و دعم RTL و Dark Mode و Arabic.
فكرة Multi-backend Storage Adapter قابلة للتطوير لاحقاً.

٤.٣ خطوات التكامل ثنائي الاتجاه
‏Planner — Two-way Sync
‏Subscribe على Graph change notifications للـ Planner tasks (POST /subscriptions).
‏Event Grid يوجّه الـ webhook للـ Azure Function: on-planner-change.
‏Function تجلب التغيير الكامل، تحدّث Azure SQL، ثم تبث إشعار SignalR للعملاء.
في الاتجاه العكسي: UI → BFF → POST /planner/tasks/{id} + Update Azure SQL في نفس المعاملة.
‏D365 Project Accounting — Two-way Sync
للـ inbound: Timer-based Azure Function كل 5 دقائق → D365 OData $filter=modifiedDateTime gt LAST_SYNC.
‏Upsert في Azure SQL مع حقل externalETag.
للـ outbound: UI change → BFF → Validation → D365 OData PATCH + Azure SQL commit في transaction واحد.
استخدام Batch API ($batch) لتقليل round-trips عند Sync Full.

٨. الخاتمة و الخطوات التالية
التطبيق كأساس وظيفي يمثل رصيداً قيماً، لكنه يحتاج تحولاً معمارياً واضحاً ليصبح منتجاً مؤسسياً صالحاً للاستخدام على مستوى الشركة. الخطة الجديدة المرفقة في المستند (Project_Plan_Sprints.docx) تُنظّم العمل على 12 سبرنت أسبوعي (~3 أشهر) تبدأ بإصلاح الأساس الأمني و الخوارزمي، ثم بناء Azure BFF، ثم تكامل Planner و D365 ثنائي الاتجاه، وتنتهي بـ Teams App production-ready.
الخطوات التالية الفورية:
الحصول على Azure subscription و Entra Admin access و D365 Sandbox مع صلاحيات API.
تعيين فريق مُصغّر: Tech Lead + 2 Devs + DevOps + QA + PM (≈ 5 أشخاص).
كِك-أوف Sprint 1 في غضون أسبوع — إصلاح الأمان الحرج و Critical Path bugs.
إعداد Backlog في Azure DevOps/GitHub Projects من الملف المرفق Sprint_Backlog.xlsx.
